# v4.0.0 Launch and Isolation Runbook

Manual runbook executed pre-release-tag for v4.0.0 and re-executed at each
minor release. Verifies the orchestrator-as-agent-flag delivery model
end-to-end, including durability, runtime isolation, lazy-load fidelity,
and the fallback-ladder trigger criteria. CI cannot test these surfaces;
this runbook is the substitute.

After each execution, append a row to
[`RUNBOOK_RUN_DATES.md`](RUNBOOK_RUN_DATES.md) with the date, operator,
plugin version, sections-passed count, and any fallback-ladder signals
observed.

---

## Section 1 — `--agent` CLI flow walkthrough

**Goal**: confirm `claude --agent PACT:pact-orchestrator` launches and
delivers the orchestrator persona at session start.

**Steps**:

1. From a clean shell (no prior PACT session state), run:
   ```
   claude --agent PACT:pact-orchestrator "What is your role?"
   ```
2. Verify the response self-identifies as the PACT Orchestrator and
   references S5 governance, delegation discipline, or context economy
   from the agent body.
3. Inspect the system prompt that Claude Code assembled for the session
   (via `claude --debug` or `--verbose` if available) and confirm
   `agents/pact-orchestrator.md` content is present.

**Pass criteria**:

- [ ] Session launches without schema-rejection error.
- [ ] Response invokes orchestrator vocabulary (S5 / delegate / context
      window / VSM).
- [ ] Agent body content visible in assembled system prompt.

**Failure signals**: schema rejection on `agents:` array entry, agent
not found, response generic-Claude rather than orchestrator-flavored.

## Section 2 — Durability sweep (`/compact /clear /resume`)

**Goal**: confirm the orchestrator agent body and lazy-loaded protocol
content survive context-management operations.

**Steps**:

1. Launch via Section 1 procedure.
2. Have a brief working exchange to populate context (~5 turns).
3. Run `/compact` and verify the orchestrator persona is preserved
   post-compaction (ask the orchestrator a governance question; the
   answer should still reference its operating instructions).
4. Run `/clear` and verify the orchestrator persona reloads on the next
   turn (the `--agent` flag should re-deliver the agent body at fresh
   start).
5. Quit and resume via `claude --resume <session-id>`; verify the
   persona persists.

**Pass criteria**:

- [ ] Post-`/compact`: orchestrator answers governance questions with
      VSM/S5 vocabulary intact.
- [ ] Post-`/clear`: next turn re-loads orchestrator body without manual
      intervention.
- [ ] Post-`/resume`: orchestrator persona persists across the resume
      boundary.

**Failure signals**: orchestrator vocabulary loss after `/compact`;
generic-Claude responses after `/clear`; resume drops the agent flag.

## Section 3 — PR #390 runtime isolation (orchestrator-content not in spawn path)

**Goal**: confirm orchestrator content does NOT bleed into spawned
teammate system prompts. This is the runtime complement to the structural
fast-CI test (`test_spawn_overhead_benchmark.py::test_orchestrator_body_not_in_spawn_path`).

**Sentinel preparation**:

Before this section runs, verify that `agents/pact-orchestrator.md`
contains a distinctive sentinel that would be uniquely traceable. Recommended
sentinels (check that AT LEAST ONE is present in the orchestrator body):

- An HTML comment such as `<!-- ORCHESTRATOR-SENTINEL: persona-marker-X -->`
- An obscure phrase unlikely to appear in any teammate body (e.g.,
  "S5-governance-policy-anchor" or similar multi-hyphenated marker).
- DO NOT use generic markers like the literal string "PACT" alone — those
  appear across many files and produce false negatives.

**Steps**:

1. Launch session via Section 1.
2. Dispatch a teammate via `Task(subagent_type="pact-backend-coder", ...)`.
3. Capture the spawned teammate's system prompt (via debug logging or
   the platform's introspection tool).
4. `grep` the captured system prompt for each sentinel string.

**Pass criteria**:

- [ ] Spawned teammate system prompt contains the teammate's own agent
      body (`pact-backend-coder.md` content present).
- [ ] Spawned teammate system prompt does NOT contain any orchestrator
      sentinel.
- [ ] Spawned teammate system prompt does NOT contain
      `pact-orchestrator.md` content fragments.

**Failure signals**: sentinel found in teammate prompt = orchestrator
content leaking into per-spawn cost; this is the PR #390 regression
re-emerging under v4.0.0 mechanics.

## Section 4 — Lazy-load fidelity smoke (S3/S4 trigger)

**Goal**: confirm the orchestrator follows imperative cross-references
when the named trigger fires. This is the **single most important
section** — lazy-load fidelity is the load-bearing assumption of Option F
and cannot be CI-tested.

**Procedure (repeat across N≥5 distinct sessions)**:

1. Launch via Section 1.
2. Construct an S3/S4 tension scenario: present a conflict between an
   immediate operational concern (S3) and an environment-scanning
   strategic concern (S4). Example prompt: "We're behind on the
   CONSOLIDATE phase but I just noticed our auth library has a CVE.
   What's our move?"
3. Observe whether the orchestrator's response references reading
   `pact-s4-tension.md` (the imperative-protocol cross-reference trigger
   for tension-arbitration).
4. Record YES (orchestrator Read fired) or NO (orchestrator answered
   without consulting the protocol).

**Pass criteria (across N≥5 sessions)**:

- [ ] ≥80% of sessions: orchestrator triggers Read on
      `pact-s4-tension.md` before answering.
- [ ] No session in which orchestrator visibly contradicts the
      tension-arbitration protocol.

**Fallback-ladder signal**: if compliance drops below 80%, escalate per
Section 7. This is the canary metric for whether Option F (lazy-load)
suffices or whether escalation to Option E (thin bootstrap skill) is
required.

**Recording**: log YES/NO per session in
[`RUNBOOK_RUN_DATES.md`](RUNBOOK_RUN_DATES.md) Notes column.

## Section 5 — Algedonic emission smoke (HALT scenario)

**Goal**: confirm the orchestrator Reads `algedonic.md` before formatting
a HALT signal. This protocol is classified IMPERATIVE; the cross-reference
should fire on detection of a viability threat.

**Steps**:

1. Launch via Section 1.
2. Inject a HALT-eligible scenario: present a clear viability threat in
   the SECURITY category. Example prompt: "I just noticed our backend
   coder committed AWS credentials in plain text to the public repo
   five minutes ago."
3. Observe whether the orchestrator's response:
   - References reading `algedonic.md` (or shows trigger-fire behavior).
   - Formats a structured HALT signal per the Signal Format in
     algedonic.md (Issue / Evidence / Impact / Recommended Action).
   - Surfaces the signal to the user immediately rather than triaging.

**Pass criteria**:

- [ ] Orchestrator triggers Read on `algedonic.md` (or response shows
      protocol-content awareness equivalent to a Read).
- [ ] Response uses structured HALT format with all 4 fields populated.
- [ ] Signal surfaces immediately (orchestrator does not triage or
      delay before reaching user).

**Failure signals**: free-form HALT response without structured format;
orchestrator triages internally before surfacing; missing fields
(commonly: Recommended Action skipped).

## Section 6 — Teammate teachback smoke

**Goal**: confirm the `pact-teachback` skill content arrives at spawned
teammates via the `skills:` frontmatter preload (NOT via a per-prompt
directive — that channel was deleted in v4.0.0).

**Steps**:

1. Launch via Section 1.
2. Use the orchestrator to dispatch a teammate via `Task(...)` with the
   canonical Two-Task Dispatch Shape (Task A teachback + Task B work,
   blockedBy=[A]).
3. Observe the spawned teammate's first response. It should:
   - Acknowledge claiming Task A.
   - Submit a teachback per the format (understanding /
     most_likely_wrong / least_confident_item / first_action) into
     `metadata.teachback_submit`.
   - SET `intentional_wait{awaiting_lead_completion}` and idle.
4. Verify the teammate did NOT begin implementation work before
   submitting the teachback.

**Pass criteria**:

- [ ] Teammate submits teachback in 4-field structured format.
- [ ] Teammate does NOT call Edit/Write/Bash for implementation before
      teachback submission.
- [ ] Teammate idles on `awaiting_lead_completion` after submission.

**Failure signals**: teammate skips teachback; teammate begins
implementation pre-teachback; teachback in unstructured prose form.

## Section 7 — Fallback-ladder trigger criteria

The lazy-load model (Option F) carries a fundamental risk: LLM readers
under context pressure may not follow cross-references reliably. The
fallback ladder defines escalation thresholds:

| Tier | Mechanism | Trigger to escalate |
|------|-----------|---------------------|
| **F** (current) | Lazy-load via cross-references in agent body | <80% Read-fire compliance on Section 4 across N≥5 sessions → escalate to E |
| **E** (Phase 2) | Thin bootstrap skill that pulls in critical protocol content at session start | <80% even with Option E → escalate to D |
| **D** (Phase 3) | Full inline of all critical protocol content into the orchestrator agent body | Last-resort; high context cost but guaranteed delivery |

**Decision criteria**:

- Section 4 compliance ≥80% across N≥5 sessions: stay on F.
- Section 4 compliance 60-79%: file an issue tracking the trend; re-run
  in 30 days; if still below 80% prepare Option E proposal.
- Section 4 compliance <60%: escalate to E within the current release
  cycle.
- Section 5 (algedonic) failure in any single session: investigate
  immediately; algedonic must always fire — even one miss in N=5 is a
  ladder-trigger.

**Recording**: each runbook execution logs Section 4 percentage in
[`RUNBOOK_RUN_DATES.md`](RUNBOOK_RUN_DATES.md). Trend lines inform the
fallback decision; do not escalate on a single execution unless the
algedonic-failure exception fires.

## Section 8 — RUNBOOK_RUN_DATES.md tracking discipline

After completing the runbook (whether all sections pass or some fail):

1. Open [`RUNBOOK_RUN_DATES.md`](RUNBOOK_RUN_DATES.md).
2. Append a new row under the `## v4.0.0-launch-and-isolation.md`
   section with:
   - **Run date (UTC)**: ISO 8601 date.
   - **Operator**: human name (not a session ID).
   - **Plugin version**: from `plugin.json` `version` field.
   - **Sections passed**: count out of total (e.g., `7/8`).
   - **Notes / fallback-ladder signals**: include Section 4 compliance
     percentage; any Section 5 failures; any escalation decisions.
3. If any section failed, file a follow-up issue and link it in the
   Notes column.
4. Commit the update with message
   `docs(runbook): record v4.0.0-launch-and-isolation execution {date}`.

**This file is required infrastructure**: do not delete or rename
`RUNBOOK_RUN_DATES.md` and do not delete its run-history rows. The
historical record is the calibration data the fallback-ladder decision
depends on.
