# Implementation Plan: Redesign pact-memory Architecture

> Generated by `/PACT:plan-mode` on 2026-03-13
> Status: IMPLEMENTED

## Summary

Redesign the pact-memory system from a monolithic Memory Lifecycle Protocol (MLP) to a clean two-mechanism split: automatic lightweight retrieval via SubagentStart hook injection, and orchestrator-driven deferred saves via the existing pact-memory skill. Memory saves become a structural part of the task hierarchy — every agent's work automatically includes a "save memory" task, blocked until work stabilizes. Remove the dedicated memory agent — all memory operations become skill-based.

The hook fires automatically at spawn and injects via `additionalContext`, combining with any existing SubagentStart hook output (e.g., peer_inject.py).

#### Design Approach

**Two-mechanism split with structural save tasks:**

```
┌──────────────────────────────────────────────────────┐
│                   ORCHESTRATOR                        │
│                                                       │
│  CLAUDE.md: Memory Save Workflow                      │
│  - Creates save tasks as part of standard QDCL        │
│  - Save tasks blocked until work stabilizes           │
│  - Unblocks save tasks → messages agents to save      │
│  - Agent loads Skill("pact-memory") + saves           │
└───────────────┬────────────────┬─────────────────────┘
                │                │
        ┌───────┴──────┐  ┌─────┴──────────┐
        │ HANDOFF GATE │  │ MEMORY RETRIEVE │
        │ TaskCompleted│  │ SubagentStart   │
        │              │  │                 │
        │ Explicit:    │  │ Injects ~20-30  │
        │ "Create save │  │ lines retrieval │
        │  task for X" │  │ instructions    │
        └──────────────┘  └─────┬───────────┘
                                │
                ┌───────────────┴──────────────┐
                │      SPECIALIST AGENTS        │
                │                               │
                │  At spawn: retrieval injected  │
                │  During work: search memory    │
                │  At handoff: save_hints +      │
                │    memory_saved: false         │
                │  On save task: Skill("pact-    │
                │    memory") → save → true      │
                └───────────────────────────────┘
```

**Task Hierarchy (structural save tasks):**

Memory saves are not an afterthought — they are a standard part of the task hierarchy. The orchestrator creates save tasks as part of its QDCL for every agent, blocked until work stabilizes.

```
[Feature] "Add auth endpoints"
├── [Agent]  "backend-coder: implement auth"       ✅ completed
├── [Agent]  "test-engineer: test auth"            ✅ completed
├── [Save]   "backend-coder: save memory"          ⏳ blocked until stable
└── [Save]   "test-engineer: save memory"          ⏳ blocked until stable
```

Save tasks do NOT block the feature task — they are tracked alongside it but are
best-effort. The feature can complete (PR, merge) without saves finishing. This keeps
the critical path clean while ensuring saves are tracked and not forgotten.

**Lifecycle:**
1. Orchestrator creates agent work task (standard QDCL)
2. Orchestrator creates corresponding save task, blocked until stable
3. Agent completes work → handoff includes `memory_saved: false`
4. Handoff gate fires explicit instruction: "📋 ACTION REQUIRED: Create memory save task for {agent_name} (task #{id}) if not already tracked."
5. Work stabilizes (bugs fixed, tests pass, reviews clean)
6. Orchestrator unblocks save tasks → messages each idle agent
7. Agent loads `Skill("pact-memory")`, saves learnings, updates `memory_saved: true`

#### Key Decisions

| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Retrieval mechanism | A) Protocol file B) Inline in agents C) SubagentStart hook | C) Hook | Auto-loaded, reliable, single source of truth, ~500-800 tokens |
| Save trigger | A) Agent self-saves B) Orchestrator-driven C) Structural save tasks | C) Structural | Save tasks are part of QDCL task hierarchy; blocked until stable; handoff gate is safety net |
| Memory agent | A) Keep B) Remove | B) Remove | User directive; skills-only model is simpler |
| Skill structure | A) Split read/write B) Keep unified | B) Unified | One skill is easier to maintain; loaded on-demand only |
| save_hints in handoff | A) New field B) Use existing handoff | B) Existing | Handoff already has produced, decisions, uncertainty, etc. — no new fields needed |
| Hook placement | A) Extend peer_inject B) Separate hook | B) Separate | Single responsibility; hooks are fast Python scripts |
| Handoff field name | `memory_used` → `memory_saved` | `memory_saved` | Reflects deferred model — memory not "used" at task time |

#### Interface Contracts

**1. SubagentStart hook → Agent context (retrieval injection)**
```
additionalContext: |
  ## Memory Retrieval (Automatic)
  Search for prior context relevant to your task before starting work.

  ```bash
  cd ~/.claude/pact-memory && python -m scripts.cli search "{your domain} {task topic}"
  ```

  Include results in your teachback as:
  MEMORY REPORT:
  - Searched for: "{query}"
  - Found: {N} relevant memories
  - Key context: {summary or "None — starting fresh"}
```

**2. Agent handoff → Task metadata (existing structure, no new fields)**
```json
{
  "handoff": { "produced": [...], "decisions": [...], "uncertainty": [...], ... },
  "memory_saved": false
}
```
The existing handoff fields (produced, decisions, uncertainty, integration, open_questions)
provide sufficient material for constructing memory saves. No save_hints field needed.

**3. Handoff gate → Orchestrator (explicit action)**
```
stdout: "📋 ACTION REQUIRED: Create memory save task for {agent_name} (task #{id})
if not already tracked. Agent completed without saving to pact-memory.
Add save task to task list, blocked until work stabilizes."
exit code: 0 (non-blocking)
```

This is a safety net — in the normal flow, the orchestrator proactively creates save tasks
as part of QDCL. The handoff gate catches cases where a save task wasn't created.

**4. Orchestrator QDCL → Save task creation**

When creating an agent's work task, the orchestrator also creates a save task:
```
TaskCreate(subject="{agent-name}: save memory", description="Load Skill('pact-memory')
and save learnings from task #{work_task_id}. Use save_hints from task metadata.
After saving, TaskUpdate(taskId, metadata={'memory_saved': true, 'memory_id': '{id}'}).")
TaskUpdate(save_task_id, addBlockedBy=["work-stabilized-gate"])
```

**5. Orchestrator → Agent (unblock + save instruction)**

When work stabilizes, orchestrator unblocks save tasks and messages each idle agent:
```
SendMessage: "Your save task is unblocked. Load Skill('pact-memory') and save your
learnings from this session. Use the save_hints you included in your handoff.
After saving, TaskUpdate(taskId, metadata={'memory_saved': true, 'memory_id': '{id}'})."
```

## TL;DR

ok so here's what we're doing

the old model was bloated — every agent had a 28-line memory lifecycle block copy-pasted into it, they were told to load a heavy skill (~1500 tokens) every time, save memory mid-task (while things are still in flux), and we had a whole dedicated memory agent just for saves. it was overkill and unreliable (agents would forget to read protocols or load the skill).

new model is two clean mechanisms:

**retrieval** — a small SubagentStart hook (`memory_retrieve.py`) that fires automatically when any PACT agent spawns. injects ~20-30 lines of "search for prior context" instructions into their context. no skill load, no protocol file to read, no hoping the agent remembers. it just shows up in their context. ~500 tokens. done.

**saves** — deferred and orchestrator-driven. agents do their work, hand off with `memory_saved: false`, and move on. the orchestrator creates a "save memory" task for each agent as part of its normal task tracking — it's structural, not an afterthought. those save tasks stay blocked until the work stabilizes (bugs fixed, tests pass, reviews clean). then the orchestrator unblocks them, messages the idle agents, they load the full pact-memory skill and save their learnings. saves don't block the feature — they're best-effort alongside it.

the handoff gate becomes a safety net: if the orchestrator somehow didn't create a save task, the gate fires an explicit "hey, create a save task for this agent" message. not a vague warning, an action item.

what we're deleting: the memory agent (skills only now), the MLP protocol file (hook replaces it), `memory_enforce.py` (forces saves at shutdown, contradicts the whole deferred model), and all the inline MLP references scattered across 10 agent files and 5 command files.

net result: agents get reliable memory retrieval at ~1/3 the token cost, saves happen at the right time instead of mid-churn, and we cut a ton of duplicated instructions.
